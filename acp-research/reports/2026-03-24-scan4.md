### 新增 Features（对 ACP 的借鉴价值）

- **SDK 向后兼容机制**（[#1401](https://github.com/a2aproject/A2A/issues/1401)）：提供 SDK 在版本升级时向后兼容的能力。**建议 ACP v1.x 跟进**——Python/Node SDK 应明确注明最低兼容版本。

### 结论：A2A 走向企业复杂度，ACP 定位更清晰

A2A v1.0.0 的每一个 breaking change 都在往"企业级"方向走（OAuth、gRPC 多租户、复杂推送通知）。
ACP 的差异化优势（P2P、无 OAuth、单文件部署）**反而更突出**。

## 📋 A2A Issue #1667（Heartbeat Agent）— 仍在讨论

**问题核心**：A2A 假设 Agent 是常驻服务，无法描述 cron 型 Agent 的可用性。

提出的 Extension 方案：
```json
{
  "extensions": [{
    "uri": "https://example.com/extensions/availability/v1",
    "params": {
      "scheduleType": "cron",
      "cronExpression": "0 */4 * * *",
      "nextActiveAt": "2026-03-21T04:00:00Z",
      "taskLatencyMaxSeconds": 14400
    }
  }]
}
```

**ACP 现状**：已在 `/.well-known/acp.json` 原生支持 `availability` 块 + `PATCH` API（v1.2，commit `c10c230`）。  
**结论**：ACP 领先 A2A 至少 1 个版本周期。`ROADMAP.md` 描述准确，无需更新。

## 📋 A2A Issue #1672（Agent Identity）— 格局变化

**新发现**：#1672 不再是讨论"是否需要"，而是已有人提交 **参考实现**（getagentid.dev）：
- ECDSA P-256 证书，每个 Agent 独立签发
- 验证 API（实时凭据校验）
- 已集成 CrewAI、LangChain、MCP

**提案的 AgentCard 字段**：
```json
{
  "verifiedIdentity": {
    "agentId": "agent_abc123",
    "certificate": "<base64-encoded certificate>",
    "issuer": "getagentid.dev",
    "verificationEndpoint": "https://getagentid.dev/api/v1/verify"
  }
}
```

**ACP 现状（v1.5 ed25519+ca 混合模型 vs A2A 提案）**：

| 维度 | ACP v1.5 | A2A 提案 |
|------|----------|----------|
| 密钥算法 | Ed25519 | ECDSA P-256 |
| 自主权 | ✅ `did:acp:` — 无需第三方颁发 | ❌ 依赖 getagentid.dev 中心化 CA |
| CA 支持 | ✅ 可选 `--ca-cert`（任意 CA） | ✅ getagentid.dev 统一颁发 |
| 离线验证 | ✅ 自主权模式可完全离线验证 | ❌ 需访问 `verificationEndpoint` |
| 中心化风险 | 低（CA 可选，不强制依赖） | 高（单一 CA + 验证端点，单点故障） |

**结论**：ACP v1.5 设计更优（自主权优先 + 可选 CA），且 Ed25519 比 ECDSA P-256 更高效。  
**建议**：在 README / 文档中明确指出 ACP 不依赖第三方 CA，这是差异化卖点。

```
ACP 在 heartbeat/cron agent、P2P NAT 穿透两个维度仍领先。
```
