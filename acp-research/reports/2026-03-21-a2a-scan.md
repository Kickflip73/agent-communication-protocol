## 关键发现 1：A2A #883 — 意外的 SQLAlchemy 强制依赖

**Issue：** [Bug]: The SDK has a SQL Alchemy dependency even when not using any SQL capability  
**日期：** 2026-03-20

**问题描述：**  
A2A SDK v1.0.0-alpha.0 中，仅 `from a2a.server.request_handlers import RequestHandler` 就会
触发 `sqlalchemy` 依赖链：
```
request_handlers → response_helpers → compat.v0_3.conversions → server.models → sqlalchemy
```
即使用户完全不使用 SQL 功能，也必须安装 `a2a-sdk[http-server,sql]` 才能运行。

**对 ACP 的启示：**  
这正是 ACP 单文件设计的优势所在。`acp_relay.py` 唯一的强依赖是 `websockets`，
所有可选功能（mDNS、Ed25519、HMAC）都在 try/except 块中优雅降级。
- ACP 的 `pip install cryptography` 是**真可选**——不安装也能正常运行
- A2A 的模块耦合导致看似可选的功能产生了强制依赖

**行动：** 维持 ACP 零隐式依赖原则；在 README 中明确标注哪些是可选 pip 包。

## 关键发现 2：A2A #876 — REQUIRED 字段静默接受（不校验）

**Issue：** SDK doesn't validate proto REQUIRED fields in SendMessage  
**日期：** 2026-03-19

**问题描述：**  
A2A v1.0 Python SDK 不校验 `SendMessage` 中的 `REQUIRED` proto 字段。
缺少 `messageId`、`role`、`parts` 时不报错，静默处理。
按规范应返回 `InvalidParamsError (-32602)`。

**对 ACP 的启示：**  
ACP 的 `/message:send` 已有明确校验（`ERR_INVALID_REQUEST` on missing parts/text），
但尚未强制要求 `message_id`（当前是自动生成）。

这实际上是 ACP 的**有意设计**：
- 客户端可选提供 `message_id`（自带幂等性控制）
- 未提供时服务端自动生成（便于快速接入）
- 区别于 A2A 的 REQUIRED + 不校验 = 最坏情况

值得在 spec 中明确说明这个设计决策，避免外部开发者困惑。

**行动：** 在 `spec/core-v0.8.md`（v0.9 任务）中增加「message_id 生成策略」设计说明章节。
