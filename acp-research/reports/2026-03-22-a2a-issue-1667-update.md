## Issue 核心内容摘要

作者描述了 A2A spec 的一个盲点：**Heartbeat-based agents**（定期唤醒型，如每 4 小时一次的 cron agent）在 A2A 中没有任何支持。

### A2A 缺失的字段（作者列举）

| 字段 | 说明 |
|------|------|
| `scheduleType` | cron / interval / persistent / manual |
| `nextActiveAt` | 下次活跃时间 |
| `lastActiveAt` | 上次活跃时间 |
| `taskLatencyMaxSeconds` | 最大任务延迟（心跳间隔决定，非网络 RTT） |

作者建议通过 A2A Extension 机制表达：

```json
{
  "extensions": [{
    "uri": "https://example.com/extensions/availability/v1",
    "params": {
      "scheduleType":          "cron",
      "cronExpression":        "0 */4 * * *",
      "timezone":              "America/New_York",
      "nextActiveAt":          "2026-03-21T04:00:00Z",
      "lastActiveAt":          "2026-03-21T00:00:00Z",
      "taskLatencyMaxSeconds": 14400
    }
  }]
}
```

### 作者的问题

1. 这是 A2A 的已知盲点吗？
2. Extension 机制是正确路径，还是应该进入 AgentCard 核心 schema？
3. 如何处理 tasks/send 到离线 Agent 的情况？（relay/proxy？queue？）
4. A2A 生态有没有 relay/proxy 模式的先例？

### 关键洞察

**ACP 的 `availability` 块不是 Extension，是 AgentCard 核心字段**。

作者的问题之一是"是进核心还是进 Extension？" — ACP 选择了核心，因为这是 heartbeat agent 的基础能力，不是可选扩展。这个决策是对的。

作者还问："A2A 生态有没有 relay/proxy 的先例？" — ACP 整个存在的意义就是这个模式。ACP Relay 是专门为 heartbeat/cron agent 设计的通信层，任务在 agent 离线时排队，唤醒时处理。
