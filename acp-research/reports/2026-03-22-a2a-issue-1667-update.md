# A2A Issue #1667 深度分析（2026-03-22 更新）

**原始 issue**：[google-a2a/A2A #1667](https://github.com/a2aproject/A2A/issues/1667)  
**标题**：Protocol Question: Heartbeat-based agents: availability metadata and offline-first task handling  
**状态**：OPEN，无官方回复（截至 2026-03-22 14:23 CST）

---

## Issue 核心诉求

提问者识别出 A2A 的架构假设与实际部署模式之间的差距：

> "A2A spec assumes a live service endpoint... But this assumes the initial `tasks/send` POST reaches a live server. For heartbeat agents, there's no server listening between cycles."

提问者列出 AgentCard 缺失的字段：
- `scheduleType`（cron / interval / persistent / manual）
- `nextActiveAt` / `lastActiveAt`
- `taskLatencyMaxSeconds`
- 批处理 vs 即时处理的区分

他们建议的解法是通过 A2A **Extension 机制**：
```json
{
  "extensions": [{
    "uri": "https://example.com/extensions/availability/v1",
    "params": {
      "scheduleType": "cron",
      "cronExpression": "0 */4 * * *",
      "nextActiveAt": "2026-03-21T04:00:00Z",
      "taskLatencyMaxSeconds": 14400
    }
  }]
}
```

---

## ACP 对比优势

| 维度 | A2A（现状） | ACP v1.2（已实现）|
|------|-------------|-------------------|
| 调度元数据 | 无原生支持，需用 Extension hack | ✅ AgentCard 原生 `availability` 块 |
| 实时更新 | 无（需重启服务） | ✅ `PATCH /.well-known/acp.json` |
| 字段标准化 | 草案阶段，uri 由各方自定义 | ✅ 标准字段：mode/interval_seconds/next_active_at/last_active_at/task_latency_max_seconds |
| 向后兼容 | 取决于 Extension 规范实现 | ✅ 全部字段可选，v1.0 兼容 |

---

## 战略意义

1. **A2A 社区在问的问题，ACP 已经有答案** — issue 提出的所有字段 (`scheduleType`→`mode`, `nextActiveAt`, `lastActiveAt`, `taskLatencyMaxSeconds`) 在 ACP v1.2 中已标准化
2. **Extension hack vs 原生支持** — A2A 走 Extension 路线意味着碎片化（每家定义自己的 uri）；ACP 是标准字段，互操作性更强
3. **PATCH API 是 ACP 独有的** — A2A 即使实现了 Extension，也没有机制让运行中的 Agent 不重启就更新调度元数据
