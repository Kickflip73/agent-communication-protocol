## Theme: Ecosystem & Optional Identity

v0.8 shifts from protocol definition (v0.5–v0.7) to **ecosystem expansion**: more SDKs, a compatibility test suite, and an optional Ed25519 identity extension that individuals can adopt without any PKI overhead.

---

## Feature Backlog

### P0 — Must Have

#### 1. Node.js SDK (`sdk/node/`)

A Node.js client mirroring the Python `RelayClient` API surface.

**Rationale:** Most frontend/serverless agents run on Node. Without a Node SDK, ACP adoption is blocked for the largest Agent runtime ecosystem.

**Scope:**
- `RelayClient` (sync-like via async/await) — same methods as Python SDK
- `send()`, `recv()`, `stream()` (EventSource), `peers()`, `tasks()`, `query_skills()`
- Zero external dependencies (built-in `fetch` + `EventSource`)
- TypeScript types included
- 15+ unit tests (mock HTTP server)

**Target file:** `sdk/node/acp_sdk/relay_client.js` + `relay_client.d.ts`

---

#### 2. Compatibility Test Suite (`tests/compat/`)

A protocol-level test suite that validates any ACP implementation (not just `acp_relay.py`) against the spec.

**Rationale:** Without a shared test suite, "ACP compatible" is a claim, not a property. The suite makes interoperability verifiable.

**Scope:**
- Black-box HTTP tests (pytest + httpx)
- Covers: AgentCard schema, `/message:send`, `/stream` SSE, `/tasks`, `/peers`, error codes
- Parameterized by `ACP_BASE_URL` — run against any implementation
- CI-friendly (exit code 0/1)
- README with "how to certify your agent"

**Target file:** `tests/compat/test_acp_compat.py`

---

### P1 — Should Have

#### 3. Ed25519 Optional Identity Extension (`spec/identity-v0.8.md`)

An opt-in identity scheme for agents that need cryptographic message attribution beyond HMAC (v0.7).

**Rationale:** A2A #1575 (Agent Passport System) shows real user demand for verifiable identity. ACP's answer: make it a clean, optional extension with zero mandatory PKI.

**Design constraints (non-negotiable):**
- ❌ No certificate authorities
- ❌ No mandatory key registration
- ❌ No revocation infrastructure (cascade revocation = APS's problem, not ours)
- ✅ Self-sovereign: agent generates keypair, publishes public key in AgentCard
- ✅ Optional: agents without `identity.scheme = "ed25519"` in AgentCard are unaffected
- ✅ Upgrade path from HMAC (v0.7 `sig`) to Ed25519 without breaking changes

**Wire format:**
```json
{
  "type": "acp.message",
  "message_id": "msg_abc123",
  "identity": {
    "scheme": "ed25519",
    "public_key": "<base64url-encoded 32-byte public key>",
    "sig": "<base64url-encoded 64-byte Ed25519 signature>"
  },
  ...
}
```

**Signature input:** canonical JSON of envelope fields excluding `identity.sig` itself.

**AgentCard declaration:**
```json
{
  "capabilities": {
    "identity": "ed25519"
  },
  "identity": {
    "scheme": "ed25519",
    "public_key": "<base64url>"
  }
}
```

**Reference:** `aeoess/agent-passport-system` (A2A #1575) for cross-language canonical JSON approach.

## Design Decisions

### Ed25519 vs HMAC coexistence

Agents may declare both `hmac_signing: true` and `identity: "ed25519"` — they serve different use cases:

| | HMAC (v0.7) | Ed25519 (v0.8) |
|--|-------------|----------------|
| **Setup** | Shared secret (pre-agreed) | Self-sovereign keypair |
| **Verifiable by** | Only parties who know the secret | Anyone with public key |
| **Use case** | Closed deployments (home lab, team) | Open federation, audit trails |
| **Overhead** | ~1ms HMAC-SHA256 | ~2ms Ed25519 sign/verify |

### No key revocation in v0.8

APS's cascade revocation is powerful but complex. ACP v0.8 defers revocation entirely:
- Agents rotate keys by updating AgentCard `identity.public_key`
- Receivers SHOULD re-fetch AgentCard on key mismatch
- Formal revocation (CRL / OCSP-style) = v1.0 candidate, only if demanded

### Compat test suite scope boundary

The compat suite tests **protocol compliance**, not performance or security hardening. It will not test:
- Throughput / latency benchmarks
- HMAC/Ed25519 signature security properties
- mDNS network behavior (requires real LAN)
